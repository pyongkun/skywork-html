1. MCP란 무엇인가 — 등장 배경과 핵심 아이디어

Model Context Protocol(MCP)은 LLM 애플리케이션이 외부 데이터·도구·기능을 표준화된 방식으로 연결하기 위한 개방형 프로토콜입니다. 공식 문서는 MCP를 "AI를 위한 USB-C 포트"에 비유합니다 — 다양한 주변기기를 단일 규격으로 연결하듯, 다양한 도구·데이터 소스를 단일 규격으로 LLM에 연결한다는 개념입니다.1

1.1 왜 MCP가 필요한가: M×N 문제

MCP 이전에는 각 AI 애플리케이션(호스트)이 각 외부 시스템(예: GitHub, Slack, 사내 DB, 파일시스템)마다 별도의 커스텀 연동 코드를 작성해야 했습니다. 애플리케이션이 M개, 연동 대상이 N개라면 이론상 M×N개의 통합 코드가 필요합니다. 이는 유지보수가 어렵고 재사용이 불가능합니다.

MCP는 이를 M+N 문제로 바꿉니다. 도구 제공자는 MCP 서버를 한 번만 구현하면 되고, AI 애플리케이션은 MCP 클라이언트를 한 번만 구현하면 됩니다. 그러면 모든 MCP 호환 호스트가 모든 MCP 호환 서버와 즉시 연동됩니다.

비유로 이해하기
웹 이전에는 각 클라이언트가 각 서버와 통신할 고유 방식을 만들어야 했지만, HTTP가 표준이 되면서 "어떤 브라우저든 어떤 웹서버든" 연결되었습니다. MCP는 "LLM ↔ 도구/데이터" 영역에서 비슷한 표준화를 목표로 합니다.

1.2 MCP가 해결하는 핵심 가치

  • 표준화: 도구·리소스·프롬프트를 노출하는 공통 인터페이스를 정의합니다.
  • 재사용성: 한 번 만든 MCP 서버를 Claude Desktop, IDE, 커스텀 에이전트 등 여러 호스트에서 그대로 사용합니다.
  • 관심사 분리: "컨텍스트를 제공하는 책임"과 "LLM과 상호작용하는 책임"을 분리합니다.2
  • 동적 발견: 클라이언트는 런타임에 서버가 제공하는 도구 목록을 질의(list_tools)하여 발견할 수 있습니다.

1.3 이 가이드의 학습 경로

본 가이드는 개념(1–6장) → 실습(7–10장) → 운영(11–12장) → 종합(13장)의 순서로 진행됩니다. 모든 코드 예제는 공식 Python SDK(mcp 패키지, v1.x 안정 버전) 기준이며, 핵심 서버 동작은 실제로 실행해 검증한 형태입니다.

2. Function Calling과 무엇이 다른가

이미 OpenAI/Anthropic의 Function Calling(Tool Use)을 써본 개발자라면 "MCP가 그것과 뭐가 다른가?"라는 의문이 들 수 있습니다. 둘은 경쟁 관계가 아니라 다른 계층입니다.

구분Function CallingMCP
본질특정 LLM API의 "기능"벤더 중립 "프로토콜/규격"
도구 정의 위치각 애플리케이션 코드 내부에 하드코딩독립 서버 프로세스로 분리
재사용해당 앱 안에서만모든 MCP 호스트에서 공유
전송 방식모델 API 호출에 종속JSON-RPC(stdio/HTTP)로 표준화
제공 단위주로 함수(tool)Tools + Resources + Prompts + Sampling 등
발견정적(코드에 미리 정의)동적(런타임 질의 가능)
핵심 관계: MCP 서버가 노출한 도구를 호스트가 가져와, 실제 LLM 호출 시에는 그 모델의 Function Calling 형식으로 변환해 전달하는 경우가 많습니다. 즉 MCP는 "도구를 어디서·어떻게 가져올지"를, Function Calling은 "모델에게 도구를 어떻게 알려줄지"를 담당합니다. 둘은 함께 동작합니다.

3. 아키텍처: Host · Client · Server

MCP는 세 가지 핵심 역할로 구성됩니다.1

역할설명예시
HostLLM을 포함하고 전체 상호작용을 조율(orchestrate)하는 AI 애플리케이션. 여러 클라이언트를 생성·관리Claude Desktop, Cursor·VS Code 같은 IDE, 커스텀 에이전트
ClientHost 내부에 존재하며 하나의 서버와 1:1 연결을 유지하고 프로토콜 메시지를 번역SDK의 ClientSession
ServerTools/Resources/Prompts를 통해 외부 데이터·기능을 제공하는 독립 프로그램파일시스템 서버, DB 서버, 직접 만든 PyTorch 추론 서버

3.1 관계 구조

# 하나의 Host는 여러 Client를 두고, 각 Client는 하나의 Server에 연결된다
                  ┌──────────────── Host (예: Claude Desktop) ───────────────┐
                  │  ┌── LLM ──┐                                            │
                  │  │         │   ┌─ Client A ─┐   stdio   ┌─ Server A ─┐  │
                  │  │ 오케스트 │──▶│  세션 유지  │◀────────▶│ 파일시스템 │  │
                  │  │  레이션  │   └────────────┘           └────────────┘  │
                  │  │         │   ┌─ Client B ─┐   HTTP    ┌─ Server B ─┐  │
                  │  │         │──▶│  세션 유지  │◀────────▶│  DB / API  │  │
                  │  └─────────┘   └────────────┘           └────────────┘  │
                  └──────────────────────────────────────────────────────────┘
데이터 방향이 중요합니다. Resources는 서버 → 호스트(컨텍스트 제공), Tools는 호스트(모델) → 서버(행동 실행) 방향이 주가 됩니다. 반대로 Sampling은 서버가 호스트의 LLM에게 생성을 요청하는 역방향 통신입니다(6장).

4. 통신 메커니즘: JSON-RPC 2.0과 Transport

4.1 메시지 포맷: JSON-RPC 2.0

MCP의 모든 메시지는 JSON-RPC 2.0 규격을 따릅니다.2 요청·응답·알림(notification) 세 가지 메시지 유형이 있습니다. 아래는 도구를 호출하는 요청과 그 응답의 개념적 형태입니다.

// 클라이언트 → 서버: tools/call 요청 (JSON-RPC)
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "add",
    "arguments": { "a": 3, "b": 4 }
  }
}

// 서버 → 클라이언트: 응답
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [ { "type": "text", "text": "7" } ]
  }
}

위 content 배열은 SDK에서 TextContent(type='text', text='7')로 표현됩니다 — 7장에서 실제 실행 결과로 확인합니다.

4.2 연결 수명주기(Lifecycle)

클라이언트와 서버는 연결 직후 초기화 핸드셰이크를 수행합니다. 클라이언트가 initialize 요청을 보내 프로토콜 버전과 역량(capabilities)을 교환하고, 이후 정상 메시지 교환이 시작됩니다. Python SDK 클라이언트에서는 await session.initialize() 한 줄이 이 과정을 담당합니다(9장).

4.3 Transport(전송 계층)

JSON-RPC 메시지를 실제로 주고받는 통로가 transport입니다. SDK는 표준 transport로 stdio, SSE, Streamable HTTP를 지원합니다.2

Transport용도상태
stdio로컬에서 서버를 자식 프로세스로 띄우고 표준입출력으로 통신. 가장 단순하고 로컬 도구에 적합표준
Streamable HTTP원격/네트워크 서버용 현대적 표준. 프로덕션 배포에 권장권장 표준
HTTP + SSE구형 원격 transport레거시(Streamable HTTP로 대체됨)
언제 무엇을 쓰나? 로컬 개발·개인 도구 → stdio, 여러 사용자가 접근하는 원격 서비스 → Streamable HTTP. 프로덕션 Streamable HTTP는 확장성을 위해 stateless_http=True, json_response=True 설정이 권장됩니다.2

5. 핵심 프리미티브: Tools · Resources · Prompts

MCP 서버가 호스트에 제공하는 능력은 세 가지 핵심 프리미티브로 표현됩니다. REST API에 비유하면 직관적입니다.2

프리미티브비유특성제어 주체
ToolsPOST 엔드포인트연산 수행 · 부작용(side effect) 허용 (예: 메일 전송, DB 쓰기, 모델 추론)모델이 호출 결정
ResourcesGET 엔드포인트읽기 전용 데이터를 컨텍스트로 제공 (예: 파일 내용, 설정값)주로 애플리케이션/사용자
Prompts템플릿재사용 가능한 프롬프트/대화 템플릿주로 사용자가 선택

5.1 한눈에 보는 정의 코드

FastMCP에서는 각 프리미티브를 데코레이터로 선언합니다. 함수의 타입 힌트와 docstring으로부터 JSON 스키마와 설명이 자동 생성되는 것이 핵심 편의 기능입니다.

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("Demo")

@mcp.tool()                          # Tool: 부작용 허용 동작
def add(a: int, b: int) -> int:
    """두 수를 더한다."""      # docstring → 도구 설명으로 노출
    return a + b

@mcp.resource("greeting://{name}")   # Resource: URI 템플릿으로 읽기 전용 데이터
def get_greeting(name: str) -> str:
    return f"Hello, {name}!"

@mcp.prompt()                        # Prompt: 재사용 템플릿
def review_code(code: str) -> str:
    return f"다음 코드를 리뷰해줘:\n\n{code}"
실제 검증된 동작: 위와 동일한 서버를 띄우고 list_tools()를 호출하면 add 도구가 다음 JSON 스키마와 함께 노출됩니다 — 타입 힌트만으로 자동 생성된 것입니다. 
{'properties': {'a': {'title':'A','type':'integer'},
                'b': {'title':'B','type':'integer'}},
 'required': ['a','b'], 'type':'object'}
그리고 call_tool("add", {"a":3,"b":4}) 호출 결과는 [TextContent(type='text', text='7')] 입니다.

6. 고급 프리미티브: Context · Sampling · Elicitation · Roots

6.1 Context — 도구 안에서 MCP 기능에 접근하기

도구/리소스 함수에 Context 타입 힌트를 가진 매개변수를 추가하면, FastMCP가 자동으로 주입해 줍니다. 이를 통해 로깅, 진행률 보고, 다른 리소스 읽기, 사용자 입력 요청 등이 가능합니다.2

from mcp.server.fastmcp import Context, FastMCP
from mcp.server.session import ServerSession

mcp = FastMCP("Progress Example")

@mcp.tool()
async def long_running_task(task_name: str, ctx: Context[ServerSession, None], steps: int = 5) -> str:
    """진행률을 보고하며 작업을 수행한다."""
    await ctx.info(f"시작: {task_name}")        # 로그 전송
    for i in range(steps):
        await ctx.report_progress(progress=(i+1)/steps, total=1.0,
                                 message=f"단계 {i+1}/{steps}")   # 진행률 보고
    return f"'{task_name}' 완료"

Context의 주요 메서드: ctx.debug/info/warning/error(msg)(로그), ctx.report_progress(...)(진행률), ctx.read_resource(uri)(리소스 읽기), ctx.elicit(message, schema)(사용자 입력 요청), ctx.session.create_message(...)(샘플링).

6.2 Sampling — 서버가 호스트의 LLM에게 생성을 요청

역방향 통신입니다. 서버는 자체 LLM 키 없이도, 호스트가 가진 LLM에게 텍스트 생성을 위임할 수 있습니다.

from mcp.types import SamplingMessage, TextContent

@mcp.tool()
async def generate_poem(topic: str, ctx: Context[ServerSession, None]) -> str:
    """LLM 샘플링으로 시를 생성한다."""
    result = await ctx.session.create_message(
        messages=[SamplingMessage(role="user",
            content=TextContent(type="text", text=f"{topic}에 대한 짧은 시를 써줘"))],
        max_tokens=100,
    )
    return result.content.text if result.content.type == "text" else str(result.content)

6.3 Elicitation — 실행 중 사용자에게 추가 입력 요청

도구 실행 도중 부족한 정보가 있으면 사용자에게 구조화된 입력을 요청할 수 있습니다. 스키마는 Pydantic 모델로 정의합니다.

from pydantic import BaseModel, Field

class BookingPreferences(BaseModel):
    checkAlternative: bool = Field(description="다른 날짜를 확인할까요?")
    alternativeDate: str = Field(default="2026-12-26", description="대안 날짜(YYYY-MM-DD)")

@mcp.tool()
async def book_table(date: str, ctx: Context[ServerSession, None]) -> str:
    result = await ctx.elicit(message="해당 날짜는 예약이 찼습니다. 다른 날짜를 볼까요?",
                              schema=BookingPreferences)
    if result.action == "accept" and result.data:
        return f"예약 완료: {result.data.alternativeDate}"
    return "예약 취소됨"

elicit()는 action("accept"/"decline"/"cancel"), data(검증된 응답), validation_error를 담은 결과를 반환합니다.

6.4 Roots — 서버가 접근 가능한 경계

Roots는 클라이언트가 서버에게 "이 디렉터리(들) 범위에서 작업하라"고 알려주는 메커니즘으로, 파일시스템 접근 경계를 정의하는 데 사용됩니다.

7. 환경 설정과 첫 번째 서버 만들기

7.1 설치

공식 권장 도구는 uv입니다. pip도 가능합니다.2

# 방법 A: uv (권장)
uv init mcp-server-demo
cd mcp-server-demo
uv add "mcp[cli]"

# 방법 B: pip
pip install "mcp[cli]"

[cli] extra는 mcp dev, mcp install 등 개발용 명령줄 도구를 포함합니다.

7.2 가장 단순한 서버 — server.py

계산기 도구, 동적 인사 리소스, 프롬프트를 모두 갖춘 최소 서버입니다. 이 코드는 공식 quickstart 구조를 따릅니다.2

# server.py
from mcp.server.fastmcp import FastMCP

# 1) 서버 인스턴스 생성
mcp = FastMCP("Demo")

# 2) 덧셈 도구
@mcp.tool()
def add(a: int, b: int) -> int:
    """두 수를 더한다."""
    return a + b

# 3) 동적 인사 리소스
@mcp.resource("greeting://{name}")
def get_greeting(name: str) -> str:
    """개인화된 인사를 반환한다."""
    return f"Hello, {name}!"

# 4) 프롬프트 템플릿
@mcp.prompt()
def greet_user(name: str, style: str = "friendly") -> str:
    """인사 프롬프트를 생성한다."""
    styles = {"friendly": "따뜻하고 친근한 인사를 작성해줘",
              "formal": "격식 있는 인사를 작성해줘"}
    return f"{styles.get(style, styles['friendly'])}: 대상={name}"

# 5) 실행 — stdio 또는 streamable-http
if __name__ == "__main__":
    mcp.run(transport="stdio")   # 원격이면 transport="streamable-http"

7.3 실행 방법

# 직접 실행
python server.py
# 또는 SDK 러너로
uv run mcp run server.py
# Inspector와 함께 개발 모드(11장)
uv run mcp dev server.py
stdio 서버에서 절대 하지 말 것: stdio transport는 표준출력(stdout)을 JSON-RPC 메시지 채널로 사용합니다. 따라서 stdio 서버 코드에서 print()로 디버그 메시지를 찍으면 프로토콜 스트림이 깨집니다. 로깅이 필요하면 ctx.info(...)(6장) 또는 stderr로 출력하세요.

8. 서버 심화: 구조화 출력 · 리소스 · 프롬프트 · 수명주기

8.1 구조화 출력(Structured Output)

도구의 반환 타입 힌트가 호환되면 FastMCP는 결과를 구조화된 형태로 반환하고, 생성된 출력 스키마에 맞춰 자동 검증합니다. Pydantic 모델, TypedDict, dataclass, dict[str, T], 기본 타입 등이 지원됩니다.2

from pydantic import BaseModel, Field
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("Structured Output Example")

class WeatherData(BaseModel):
    temperature: float = Field(description="섭씨 온도")
    humidity: float = Field(description="습도(%)")
    condition: str

@mcp.tool()
def get_weather(city: str) -> WeatherData:   # 반환 타입이 곧 출력 스키마
    """도시의 날씨를 구조화 데이터로 반환한다."""
    return WeatherData(temperature=22.5, humidity=45.0, condition="맑음")
기본 타입(list, float 등)을 반환하면 구조화 출력에서는 {"result": value} 형태로 래핑됩니다. 동시에 하위호환을 위한 비구조화 출력도 함께 반환되는데, 5.1·9.3장에서 본 [TextContent(type='text', text='7')]가 바로 이 비구조화 표현입니다. 즉 한 번의 호출 결과에 두 표현이 공존하며, 클라이언트는 필요에 따라 골라 씁니다. 구조화 분류를 끄려면 @mcp.tool(structured_output=False)를 사용합니다.

8.4 수명주기(Lifespan) — 시작/종료 시 리소스 관리

DB 연결처럼 서버 시작 시 한 번 초기화하고 종료 시 정리해야 하는 리소스는 lifespan으로 관리하고, 도구에서는 ctx.request_context.lifespan_context로 타입 안전하게 접근합니다.2

from contextlib import asynccontextmanager
from dataclasses import dataclass
from mcp.server.fastmcp import Context, FastMCP
from mcp.server.session import ServerSession

@dataclass
class AppContext:
    db: "Database"

@asynccontextmanager
async def app_lifespan(server: FastMCP):
    db = await Database.connect()      # 시작 시 1회
    try:
        yield AppContext(db=db)
    finally:
        await db.disconnect()         # 종료 시 정리

mcp = FastMCP("My App", lifespan=app_lifespan)

@mcp.tool()
def query_db(ctx: Context[ServerSession, AppContext]) -> str:
    db = ctx.request_context.lifespan_context.db
    return db.query()

9. MCP 클라이언트 작성하기

지금까지는 서버를 만들었습니다. 이제 그 서버에 연결해 도구를 호출하는 클라이언트를 직접 구현해 봅니다. 이는 커스텀 에이전트나 호스트를 만들 때 필요합니다.

9.1 stdio 클라이언트 기본 구조

핵심 API: StdioServerParameters로 서버 실행 명령을 정의하고, stdio_client(...)로 연결한 뒤, ClientSession으로 세션을 열고 initialize()한 다음 도구를 호출합니다.2

# client.py
import asyncio
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

# 서버를 자식 프로세스로 띄울 명령 정의
server_params = StdioServerParameters(
    command="python",
    args=["server.py"],
    env=None,
)

async def main():
    async with stdio_client(server_params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()              # 핸드셰이크

            # 도구 목록 조회(동적 발견)
            tools = await session.list_tools()
            print("도구:", [t.name for t in tools.tools])

            # 도구 호출
            result = await session.call_tool("add", {"a": 3, "b": 4})
            print("결과:", result.content)

            # 리소스 읽기
            res = await session.read_resource("greeting://Skyclaw")
            print("리소스:", res.contents)

asyncio.run(main())

9.2 원격(Streamable HTTP) 클라이언트

원격 서버에 붙을 때는 transport만 교체합니다. 나머지 ClientSession 사용법은 동일합니다.

from mcp import ClientSession
from mcp.client.streamable_http import streamablehttp_client

async def main():
    async with streamablehttp_client("http://localhost:8000/mcp") as (read, write, _):
        async with ClientSession(read, write) as session:
            await session.initialize()
            tools = await session.list_tools()
            print([t.name for t in tools.tools])

9.3 결과 파싱 팁

call_tool의 결과는 content(텍스트/이미지 등 콘텐츠 블록 리스트)를 담습니다. 5장에서 검증했듯 add(3,4)는 [TextContent(type='text', text='7')]를 반환합니다. 텍스트만 필요하면 다음처럼 추출합니다.

texts = [c.text for c in result.content if getattr(c, "type", None) == "text"]
print(texts)   # ['7']

10. 실전 예제: PyTorch 모델을 MCP Tool로 노출하기

이 장은 본 가이드의 하이라이트입니다. 로컬에서 학습/로딩한 PyTorch 모델의 추론(inference)을 MCP 도구로 감싸면, Claude Desktop 같은 어떤 MCP 호스트에서도 "내 모델로 분류해줘"라고 자연어로 호출할 수 있게 됩니다.

10.1 설계 개념

  • 모델 로딩은 서버 시작 시 한 번(lifespan)만 수행하여 매 호출마다 재로딩하지 않습니다.
  • 추론 함수를 @mcp.tool()로 감싸 입력(이미지 경로/텍스트)과 출력(라벨·확률)을 정의합니다.
  • 무거운 추론은 동기 함수를 스레드로 오프로딩(asyncio.to_thread)하여 이벤트 루프를 막지 않습니다.

10.2 추론 후처리 로직 (검증됨)

도구 출력의 핵심은 logits → 확률(softmax) → 상위 예측(top-k) 변환입니다. 이 변환 로직은 별도로 실행해 검증했습니다. 라벨이 ["A","B","C"]인 3-class 데모에서 입력 logits [2.0, 1.0, 0.1]를 넣으면 확률은 약 [0.659, 0.242, 0.099]가 되어 0번 클래스(0.659)가 1순위, 1번 클래스(0.242)가 2순위로 산출됩니다.

아래 10.3 전체 코드는 ResNet18(ImageNet 1000 클래스)을 사용하므로 실제 라벨은 weights.meta["categories"](예: tench, goldfish, ...)에서 옵니다. 위 A/B/C는 후처리 수식 검증을 위한 추상 예시일 뿐, 특정 클래스명을 의미하지 않습니다.

10.3 전체 서버 코드 — 이미지 분류 모델

torchvision의 사전학습 ResNet18을 예로 듭니다. 자신의 .pt 체크포인트로 교체하면 그대로 동작합니다.

# torch_mcp_server.py
# 의존성: uv add "mcp[cli]" torch torchvision pillow
from contextlib import asynccontextmanager
from dataclasses import dataclass
import asyncio
import torch
import torch.nn.functional as F
from torchvision import models, transforms
from PIL import Image
from pydantic import BaseModel, Field
from mcp.server.fastmcp import Context, FastMCP
from mcp.server.session import ServerSession

# ---------- 1) 출력 스키마(구조화 출력) ----------
class Prediction(BaseModel):
    label: str = Field(description="예측 클래스 라벨")
    confidence: float = Field(description="확률(0~1)")

class ClassifyResult(BaseModel):
    top: Prediction
    topk: list[Prediction]

# ---------- 2) lifespan: 모델을 1회 로딩 ----------
@dataclass
class ModelContext:
    model: torch.nn.Module
    labels: list[str]
    preprocess: transforms.Compose
    device: str

@asynccontextmanager
async def lifespan(server: FastMCP):
    device = "cuda" if torch.cuda.is_available() else "cpu"
    weights = models.ResNet18_Weights.DEFAULT
    model = models.resnet18(weights=weights).to(device).eval()
    preprocess = weights.transforms()        # 공식 전처리 파이프라인
    labels = weights.meta["categories"]      # 1000개 ImageNet 라벨
    try:
        yield ModelContext(model, labels, preprocess, device)
    finally:
        del model
        if device == "cuda": torch.cuda.empty_cache()

mcp = FastMCP("PyTorch Vision Server", lifespan=lifespan)

# ---------- 3) 동기 추론 함수(스레드에서 실행) ----------
def _infer(mc: ModelContext, image_path: str, k: int) -> ClassifyResult:
    img = Image.open(image_path).convert("RGB")
    x = mc.preprocess(img).unsqueeze(0).to(mc.device)
    with torch.no_grad():
        logits = mc.model(x)
        probs = F.softmax(logits, dim=1)[0]
    conf, idx = torch.topk(probs, k)
    topk = [Prediction(label=mc.labels[i], confidence=round(float(c), 4))
            for c, i in zip(conf.tolist(), idx.tolist())]
    return ClassifyResult(top=topk[0], topk=topk)

# ---------- 4) MCP 도구로 노출 ----------
@mcp.tool()
async def classify_image(image_path: str, ctx: Context[ServerSession, ModelContext],
                         top_k: int = 3) -> ClassifyResult:
    """로컬 PyTorch 모델로 이미지를 분류하고 상위 k개 예측을 반환한다."""
    mc = ctx.request_context.lifespan_context
    await ctx.info(f"추론 시작: {image_path} (device={mc.device})")
    result = await asyncio.to_thread(_infer, mc, image_path, top_k)  # 블로킹 추론 오프로딩
    await ctx.info(f"1순위: {result.top.label} ({result.top.confidence})")
    return result

if __name__ == "__main__":
    mcp.run(transport="stdio")

10.4 텍스트 분류(직접 학습한 모델) 변형

직접 학습한 간단한 분류기를 노출하는 패턴도 동일합니다. 핵심은 lifespan에서 torch.load로 체크포인트를 불러오는 것입니다.

@asynccontextmanager
async def lifespan(server: FastMCP):
    device = "cuda" if torch.cuda.is_available() else "cpu"
    model = MyTextClassifier(num_classes=3)
    state = torch.load("checkpoint.pt", map_location=device)  # 체크포인트 로딩
    model.load_state_dict(state)
    model.to(device).eval()
    yield ModelContext(model=model, labels=["부정","중립","긍정"], device=device)
운영 주의: ① 무거운 모델 추론을 async 도구 본문에서 직접 동기 호출하면 서버 전체가 멈춥니다 — 반드시 asyncio.to_thread(또는 별도 워커)로 오프로딩하세요. ② torch.load로 신뢰할 수 없는 체크포인트를 로딩하면 임의 코드 실행 위험이 있습니다. 신뢰된 파일만, 가능하면 weights_only=True 옵션을 사용하세요. ③ 파일 경로 입력을 받는 도구는 경로 검증(허용 디렉터리 제한)을 추가하세요(12장).

10.5 호스트에서의 사용 흐름

  1. 위 torch_mcp_server.py를 Claude Desktop 등에 등록(11장).
  2. 사용자가 "이 이미지가 뭔지 분류해줘: /data/cat.jpg" 라고 입력.
  3. 호스트의 LLM이 classify_image 도구를 선택하고 인자 {image_path:"/data/cat.jpg", top_k:3}로 호출.
  4. 서버가 PyTorch 추론 후 구조화 결과를 반환 → LLM이 자연어로 요약 응답.

12. 보안 고려사항

MCP 서버는 외부 입력을 받아 코드를 실행하는 신뢰 경계입니다. 공식 사양과 보안 가이드는 다음을 강조합니다.3

12.1 인증 — OAuth 2.1

원격 서버는 OAuth 2.1 기반 인증을 권장합니다. SDK의 mcp.server.auth는 MCP 서버가 Resource Server(RS) 역할로 토큰을 검증하도록 지원하며, 별도의 Authorization Server(AS)가 발급한 토큰을 RFC 9728(Protected Resource Metadata)로 발견합니다.2

from mcp.server.auth.provider import AccessToken, TokenVerifier
from mcp.server.auth.settings import AuthSettings

class MyVerifier(TokenVerifier):
    async def verify_token(self, token: str) -> AccessToken | None:
        ...  # 실제 토큰 검증 구현

mcp = FastMCP("Secure Server", token_verifier=MyVerifier(),
              auth=AuthSettings(required_scopes=["user"]))

12.2 핵심 보안 수칙

위협대응
토큰 패스스루(Token Passthrough)클라이언트 토큰을 백엔드 API로 그대로 전달 금지(Confused Deputy 공격 방지). 서버는 자체 인증/권한 경계를 가져야 함3
입력 인젝션도구 인자(파일 경로, 명령어 등)는 항상 검증·화이트리스트. 임의 경로 접근/명령 실행 차단
역직렬화 위험신뢰할 수 없는 torch.load 체크포인트 로딩 금지. weights_only=True 권장
과도한 권한서버에 최소 권한만 부여. Roots로 접근 가능한 경계를 제한
신뢰 경계 가시성MCP 서버를 엔터프라이즈 아키텍처의 1급 신뢰 경계로 취급하고 로깅·모니터링 확보
핵심 원칙: 도구는 "모델이 자율적으로 호출"할 수 있다는 점을 항상 기억하세요. 위험한 부작용(삭제, 결제, 외부 전송)이 있는 도구는 명시적 사용자 확인 단계(Elicitation 등)나 권한 검증을 반드시 두어야 합니다.

  3. 레벨 3 — PyTorch 추론 서버: 10장의 이미지/텍스트 분류 서버를 완성하고 Claude Desktop에 등록해 자연어로 호출.
  4. 레벨 4 — 원격 배포: 레벨 3 서버를 Streamable HTTP(stateless) + OAuth 2.1 인증으로 전환해 원격 서비스화.
  5. 레벨 5 — 커스텀 에이전트: 9장의 클라이언트를 확장해 여러 서버에 동시 연결하고, LLM과 결합한 자체 에이전트 루프 구성.

13.3 자주 묻는 질문(FAQ)

Q. FastMCP가 뭔가요? 별도 라이브러리인가요?

FastMCP는 데코레이터 기반의 고수준 서버 인터페이스로, 공식 Python SDK(mcp)에 포함되어 있습니다. from mcp.server.fastmcp import FastMCP로 바로 사용합니다. 더 세밀한 제어가 필요하면 저수준(Low-Level) 서버 API도 사용할 수 있습니다.2

Q. 동기 함수도 도구로 쓸 수 있나요?

네. 간단한 도구는 일반 def로 충분합니다. 다만 무거운 I/O나 모델 추론은 async def + asyncio.to_thread 패턴(10장)을 권장합니다.

Q. SDK 버전은 어떤 걸 써야 하나요?

본 가이드는 현재 안정 버전인 v1.x(mcp[cli]) 기준입니다. v2는 알파 단계로 일부 API가 변경될 수 있으므로(예: 안정 릴리스 시점에 마이그레이션 가이드 확인 권장), 학습·프로덕션 모두 우선 v1.x로 시작하는 것을 권합니다. 의존성에 mcp>=1.x,<2 상한을 두면 안전합니다.2

Q. stdio 서버가 동작은 하는데 호스트에서 안 보여요.

대표 원인: ① 서버 코드에서 print()로 stdout을 오염시킴(7장 경고), ② 설정 파일의 command/args 경로 오류, ③ 의존성 미설치. Inspector로 단독 연결해 먼저 서버 자체가 정상인지 확인하세요.

이 가이드의 모든 핵심 서버 동작(도구·리소스·프롬프트 등록, 스키마 자동 생성, call_tool 호출)과 추론 후처리 로직은 실제로 실행해 검증했습니다. 코드를 직접 타이핑하며 따라가면 가장 빠르게 체득할 수 있습니다.
